Meteor-React-TypeScript Todos

A conversion of the Meteor-React sample app to use TypeScript

Download this project as a .zip file Download this project as a tar.gz file

So you want to use Meteor, React, and TypeScript

Great choice! These are all fine technologies. Who wouldn't want to combine the end-to-end clarity of Meteor, the clear data flow of React, and the safety and comfort of TypeScript. You might be thinking this will be easy. After all, the official Meteor page has a tutorial for combining Meteor and React and there are established packages on Atmosphere for combining Meteor and TypeScript such as the meteortypescript compiler and typescript-libs packages, so it looks like the work has already been done for us. Alas, in programming nothing is ever this easy, and there are quite a few obstacles to overcome along the way.

A list of the problems we will need to solve:

  • We need a way to compile our .ts and .tsx files
  • We need type definitions for React and Meteor
  • Ideally, we want to use TypeScript namespaces to organize our code
  • We need a way to use the react-data-mixin with ES6-style React components
  • We need to reconcile the react-data-mixin with TypeScript typing

Each of these problems has subtleties that require a bit of thought and experimentation to solve. Below are the solutions I came up with, the result of which can be seen at the linked GitHub project.

Compiling .ts and .tsx files

I expected this step to be easy because the meteortypescript:compiler Meteor package seems like it solves this precise problem. However, meteortypescript:compiler seems only designed to convert .ts files. No matter what I tried, I could get it to correctly handle .tsx files: it would always leave the JSX elements in the compiled .js code. The documentation claims that compiler options can be given in a .tsconfig file at your project root, but even when I added the option "jsx": "react" it still incorrectly left JSX in .js files.

Sadly, this means finding our own means to compile the TypeScript files. I decided to use the Atom text editor with the atom-typescript package, although I expect these steps will be similar with any other TypeScript IDE. I used atom-typescript to generate a tsconfig.json file (on a Mac, that's Cmd+Shift+P, then "Create Tsconfig.json Project File") and set the following options:

{
    "compilerOptions": {
        "target": "es6",
        "jsx": "preserve",
        "experimentalDecorators": true,
        ...
    },
    ...
    "compileOnSave": true,
    "buildOnSave": true,
    ...
}

Now whenever we save a .ts or a .tsx file, a corresponding .js or .jsx file is automatically generated in the same directory. This does clutter up our directories somewhat, but it has benefits as well:

  • The TypeScript step of the build process is completely transparent, as we can see exactly what is being fed into Meteor and examine it if we choose to do so

  • The filesGlob option in the generated tsconfig.json automatically finds definitions for us. This eliminates the need to add reference declarations at the top of all our files, e.g.

    /// <reference path="./typings/react/react.d.ts" />

  • Since our IDE is responsible for building, we don't run into situations where the IDE insists things are correct while the build system fails or vice versa

The added .js and .jsx files are kind of annoying though, so let's see if we can hide them a bit. The first thing to do is have Git ignore them, since we don't want compiled files being committed. I create a .gitignore in my project root directory containing

**/*.js
**/*.jsx

Assuming I have already run git init, Atom will now grey out all the gitignored .js and .jsx files in its tree view. Atom also has an option to hide gitignored files entirely, but as a personal preference I prefer to be able to see them so I can poke around in them if need be.

Type definitions for React and Meteor

Type definitions for React are easy. I have the TSD definition manager installed, so from the project root I type

tsd install react --save
tsd install react-global --save

If you don't use TSD, you can just as easily fetch the definitions from DefinitelyTyped.

Now for the Meteor definitions. We could simply get these from TSD/DefinitelyTyped as well, and this is a fine solution. However, I decided to get these from the meteortypescript:typescript-libs Meteor package. This has the advantage that the definitions will update automatically with meteor update, and it also comes with a handful of other definition files for common Meteor specific libraries that don't appear on DefinitelyTyped, such as iron:router. It does take a little bit more work, so I will totally understand if you decide not to do it. But for those who want to try, we run

meteor add meteortypescript:typescript-libs

and as suggested in the documentation (after running the project) add a symbolic link to the definitions deep within the .meteor directory:

ln -s ../.meteor/local/build/programs/server/assets/packages/meteortypescript_typescript-libs/definitions typings/package_defs

However, this isn't perfect because it adds a bunch of definitions to libraries we may not be using or even have installed, like D3. Much more annoyingly, it contains a lib.d.ts file which contains the base JavaScript definitions which for some reason conflicts with TypeScript's built in definitions, causing build errors in our IDE. To solve both these problems, we'll make sure that the typings/package_defs directory is not compiled with the project by excluding it in tsconfig.json:

...
"filesGlob": [
    "**/*.ts",
    "**/*.tsx",
    "!node_modules/**",
    "!typings/package_defs/**"
],
...

Then to actually get the specific definitions we want from typings/package_defs, create a new definition file which explicitly references the desired files. I created a new file typings/package_defs.d.ts which for now just contains

/// <reference path="./package_defs/meteor.d.ts" />

Done! Try typing some React or Meteor expressions into a .ts or .tsx file now and hopefully get autocompletion and type information.

Using TypeScript namespaces

This step is optional and the solution is quite disgusting. You might decide you don't intend to use the namespace feature of TypeScript at all, in which case you can ignore this step and happily spew symbols into the global scope. But I happen to like namespaces a lot. Some advantages:

  • Exporting a single symbol into the global scope greatly reduces the possibility of naming conflicts with other libraries or future built-in functions
  • It is particularly useful when debugging because I can access all app specific names through the single global variable rather than autocompleting from the large pool of global names
  • It helps organize code by allowing simpler names in a given part of the code. For example, if I am in the namespace Todos.Task, I can declare a class named State representing the state of a Task and not worry about using up such a clear and generic name

So what makes namespaces hard to use with Meteor? We'll need to look into the implementation details of namespaces a bit. The TypeScript code

namespace Space {
    export const x = 2;
}

compiles to the JavaScript

var Space;
(function (Space) {
    Space.x = 2;
})(Space || (Space = {}));

Neat right? It grabs an existing object with the namespace name out of global scope, and then exported variables are read from and written to that object. Unfortunately, in an effort to be helpful, Meteor wraps the contents of each file in an IIFE, meaning what gets emitted actually looks like this

(function () {
    var Space;
    (function (Space) {
        Space.x = 2;
    })(Space || (Space = {}));
})();

This ruins everything, because now the var Space no longer references the global variable Space but instead declares a new one inside the scope of the newly added anonymous function. Since each reference to the namespace therefore refers to an entirely different object, the whole system comes crashing down.

Unfortunately, I cannot come up with a clean solution to this problem, so I eventually settled on the solution proposed by Dr. Llama in the post Meteor Typescript - Classes and Modules (in the linked post, Dr. Llama talks about "modules" instead of "namespaces", but they are the same concept). That is, we use eval to insert a line to make the variable refer to the global reference, e.g.

eval("var Space = this.Space || (this.Space = {})");

namespace Space {
    ...
}

Since I'll be doing this a lot, I'll factor out that string into a helper function declared in a separate file that we make sure is loaded first:

declare var namespaceEvalHack: (namespace: string) => string;

namespaceEvalHack = function namespaceEvalHack(namespace: string) {
    return `var ${namespace} = this.${namespace} || (this.${namespace} = {})`;
}

and then whenever we use namespaces we can write

eval(namespaceEvalHack("Space"));

namespace Space {
    ...
}

Whew. It's pretty ugly, but it works and now we can use namespaces to our hearts' content.

Using ReactDataMixin with ES6 Components

The Meteor react package provides a mixin called ReactDataMixin to allow the creation of React components that are reactive in the Meteor sense (i.e. update when collections update, etc.). Unfortunately, React mixins are not supported to be used with ES6 style of component declaration, which is the way supported by TypeScript. To reconcile the two, we need the help of a third-party library, react-mixin, which allows React mixins to be used with ES6 classes. react-mixin comes in two versions, one from npm and one as a standalone library that exports the global symbol reactMixin. We will choose the second one, since incorporating our app with npm is a challenge for another day (if your curious, it can be done with Meteor's cosmos:browserify package). I save the standalone lib to lib/thirdParty/react-mixin.js.

But wait, didn't we tell Git to ignore all .js files in the project? That's inconvenient. To get around this, we'll add an exception for the specific directory we decide to use for third-party libs, by adding a line to our .gitignore so it appears

**/*.js
**/*.jsx
!lib/thirdParty/**

Assuming you set "experimentalDecorators": true in your tsconfig.json as suggested above, we can now mix ReactDataMixin with our ES6 classes as follows:

@reactMixin.decorate(ReactDataMixin)
class SomeComponent extends React.Component<{}, {}> {
    ...
}

Or if you don't want to use decorators, you can do it instead with

class SomeComponent extends React.Component<{}, {}> {
    ...
}

reactMixin.onClass(SomeComponent, ReactDataMixin);

Of course, if you do this right now, TypeScript will start complaining everywhere. The typing story here is a little complicated, so let's jump into it!

Typings for ReactDataMixin

Neither ReactDataMixin nor reactMixin seem to have existing type definitions. Luckily, both are simple enough that we can write the definitions ourselves without too much trouble.

First things first: let's get a definition for ReactDataMixin. In a definition file somewhere (I used typings/react-data-mixin/react-data-mixin.d.ts), write

declare const ReactDataMixin: any;

Alright, that was easy! How about reactMixin?

declare interface ReactMixin {
    (componentPrototype: React.Component<any, any>, mixin: any): void;
    onClass: (componentClass: React.ComponentClass<any>, mixin: any) => void;
    decorate: (mixin: any) => ClassDecorator;
}

declare const reactMixin: ReactMixin;

Woo! Are we done? Not quite. When ReactDataMixin is added to a component, it adds a new field data to the host component and expects us to implement a new method getMeteorData() which reads the data from some reactive data source. Since we can't add a new field to a component with definitions alone, we will actually create a new Component superclass for this, parameterized on the type of data as well as on the types of props and state:

class MeteorComponent<P, S, D> extends React.Component<P, S> {
    public data: D;

    public getMeteorData(): D {
        throw new Error("MeteorComponent subclass must implement getMeteorData()");
    }
}

Now we can declare components which depend on Meteor reactive sources by subclassing this

@reactMixin.decorate(ReactDataMixin)
class App extends MeteorComponent<AppProps, AppState, AppData> {
    public getMeteorData(): AppData {
        ...
    }

    ...
}

And we're done! That was the last hurdle, so now we can finally develop in peace.

Other gotchas

There are a few other stumbling blocks I ran into while working on this which are too small to discuss at length but are still worth mentioning.

  • If you choose not to use namespaces, then certain type names you use may collide with third-party definitions. For example, if you naively follow the Meteor-React tutorial and use TypeScript without namespaces, then you will run into a baffling error when your class App conflicts with the type App declared in meteor.d.ts. You can avoid this problem by renaming the class to something else (say MyApp) or by using namespaces.
  • Certain Meteor API's bind some special value to this. For example, following the tutorial the Accounts package expects you use this.userId to get the current userId in a publication function argument. When writing such a function, make sure not to use ES6 arrow functions but instead to write out the full function () { ... }, since arrow functions capture the current value of this.
  • Adding types for Meteor and React code is not too bad after everything we've been through above, but if you ever get stuck, look through the linked GitHub repo for sample code.
